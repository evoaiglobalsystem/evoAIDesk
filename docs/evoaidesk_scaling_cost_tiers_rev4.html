Model update — 1 chatbot per client (revised from 3)
This document reflects the revised client model: 1 client = 3 dashboard users + 1 chatbot included free on signup. Additional bots are sold as self-serve add-ons at $5/bot/month. Baseline Firestore reads drop ~67% vs. the previous 3-bot assumption — Firebase costs fall from ~$162/mo to ~$54/mo at 100 clients. The bot workflow (Smart Version, n8n) is pre-built and production-ready; onboarding a new client's bot requires only a Telegram token and Firestore tenant registration — no custom setup.

Clients who see value in the free bot convert naturally to the $5/bot/month add-on — ~89% gross margin on each extra bot since the marginal infra cost is only ~$0.55/bot/month.

Cost optimisation levers

Option A
Extend TTL to 8 hours
Halves cache-miss Firestore reads. At the 1-bot baseline the bill is already low — but matters more as add-on bots accumulate and push read volume higher.
Saves ~40 % on Firebase reads
Option B · Recommended
24-hr TTL + Redis
Aggressive TTL combined with a $15/mo Managed Redis instance eliminates restart-spike read floods. Baseline restart spike at 100 clients is only 500 K reads — still worth it for reliability SLAs.
Best case: ~$13 → ~$80 / month
Option C · Long-term
Event-driven cache invalidation
Replace TTL entirely with change-stream or webhook-driven expiry. Cache only refreshes when lead data actually changes — eliminates unnecessary reads regardless of bot count.
Replaces TTL — max efficiency

Actions by tier — what to do and when

▶
5 clients — now
Smallest DO droplet (2 vCPU / 2 GB, $12/mo) — easily handles 5 bots with headroom for add-ons
Blaze plan required — 153 K reads/day is 3× free tier but costs only ~$3/mo
No Redis needed — restart spike is only 25 K reads ($0.015)
SQLite is fine — 5 bots means minimal write concurrency
Focus: get the free bot live within 48 hrs of signup so clients see lead capture value immediately
↑
25 clients — near term
Upgrade droplet to 2 vCPU / 4 GB ($24/mo) around 20 clients
Firebase reads only ~$14/mo — significantly better than old model; no urgency on TTL extension
Restart spike is 125 K reads ($0.075) — still cheap, monitor restart frequency
Actively promote the $5 add-on — clients actively using the free bot are the warmest upsell targets
Begin Redis planning for the 50-client milestone
⚠
50 clients — action needed
Upgrade droplet to 4 vCPU / 8 GB ($48/mo) — do this before reaching 50 clients
Add Redis ($15/mo) — 250 K restart spike + add-on bots can push this higher
Firebase baseline ~$27/mo — extend TTL to 8 hrs if add-on bots are selling well
Review Firestore security rules — tenant isolation becomes critical at this scale
SQLite still fine at 50 baseline bots — monitor if add-ons push concurrency above 80–100 bots
🔥
100 clients — optimise now
Redis non-negotiable — 500 K baseline restart spike; add-on bots can push this to 1 M+
Extend TTL to 8–24 hrs — saves $27–$45/mo at baseline, substantially more with add-ons
Consider event-driven cache invalidation to replace TTL entirely
Monitor SQLite under high add-on bot concurrency — Managed Postgres if write locking appears
Plan second n8n node if selling uptime SLAs — bot reliability becomes a paid expectation
