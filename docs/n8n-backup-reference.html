Implemented: May 2026 evoAI Internal · n8n on DigitalOcean Daily · 2:00 AM IST
Summary
Fully automated — no manual action required
Runs outside the DO server via GitHub Actions
Backups stored on Google Drive — 7 day retention
One-click restore from any date via GitHub UI
Email alert on backup failure
100% free — no billing on any service
01 Architecture overview

The backup system runs entirely outside the DigitalOcean server. GitHub Actions acts as the scheduler and orchestrator — it SSHes into the DO server, creates the backup, streams it back, and uploads it to Google Drive. The DO server itself requires no extra tools.

02 Backup flow — automatic

The backup.yml workflow triggers automatically at 2AM IST every day. n8n downtime during backup is minimal — typically under 60 seconds.

1
GitHub Actions cron fires at 2AM IST
20:30 UTC · daily
2
Runner SSHes into DO server
using DO_SSH_PRIVATE_KEY secret
3
n8n container stopped
docker stop n8n
4
Volume data tarred to /tmp
n8n-backup-YYYY-MM-DD.tar.gz
5
n8n container started back up
docker start n8n
6
Runner pulls tar via SCP
streamed to GitHub Actions runner /tmp
7
Tar deleted from DO server /tmp
keeps DO server clean
8
Runner uploads to Google Drive
via Drive API · OAuth refresh token
9
Backups older than 7 days deleted from Drive
auto-cleanup on every run
03 Restore flow — manual trigger

The restore.yml workflow is triggered manually from GitHub Actions UI. Enter the backup date and GitHub handles everything — the current live data is always preserved before overwrite.

1
Open GitHub → Actions → n8n Restore → Run workflow
enter date: YYYY-MM-DD
2
Runner downloads backup from Google Drive
by exact filename match
3
Any previous _data_before_restore_* folders deleted
keeps DO server clean
4
n8n container stopped
docker stop n8n
5
Current live _data renamed as safety backup
_data_before_restore_YYYY-MM-DD
6
Backup extracted to fresh _data folder
permissions fixed: chown -R 1000:1000
7
n8n container started
docker start n8n
8
File list printed to GitHub Actions log
visible in Actions → restore job → step details

07 Failure handling
If backup fails
GitHub automatically sends a failure email to the repository owner. No backup is created that day. The next scheduled run will attempt again at 2AM IST. You can also manually trigger backup.yml at any time from the Actions tab.

If DO server is unreachable — GitHub Actions will fail at the SSH step and send a failure notification. No data is affected.

If Google Drive upload fails — the backup tar is already downloaded to the runner. The runner will fail and notify. The DO server is unaffected and n8n remains running.

During restore — the current live data is always renamed to _data_before_restore_YYYY-MM-DD before any changes, providing a rollback path.
